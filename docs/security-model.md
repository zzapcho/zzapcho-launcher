# 보안 모델

## 핵심 전제

런처만으로는 런처 미사용 접속을 완벽하게 막을 수 없다.

클라이언트는 사용자의 PC에서 실행되므로 조작될 수 있다. 따라서 런처의 검사는 보조 방어이고, 최종 입장 차단은 서버 측에서 해야 한다.

## 1단계에서 제공하는 것

현재 단계에서는 보안 기능을 완성하지 않는다.

제공하는 기반:

- 서버 주소 고정
- 사용자 서버 입력 UI 없음
- LocalAppData 기반 데이터 폴더
- 로그 기록
- 서버 상태 확인 구조
- 이후 파일 검증/로그인/permit 기능을 붙일 수 있는 서비스 구조

## 이후 단계의 방어 계층

정식 구조는 다음 계층을 사용한다.

1. 고정 서버 전용 런처
2. 격리된 Minecraft 인스턴스 폴더
3. manifest 기반 공식 파일 목록
4. SHA-256 파일 검증
5. 알 수 없는 파일 격리
6. Microsoft 계정 인증
7. 백엔드의 짧은 입장 허가
8. Paper 서버 플러그인의 permit 검증
9. 접속 후 클라이언트 핸드셰이크

## GitHub manifest 운영

Minecraft 버전, 로더 버전, 런처 최소 버전, 공식 파일 목록은 GitHub에 있는 manifest에서 관리한다.

런처가 GitHub raw URL에서 manifest를 읽으면, 운영자는 앱을 다시 빌드하지 않고 다음 값을 바꿀 수 있다.

- Minecraft 버전
- Fabric 로더 버전
- 런처 최소 버전
- 런처 최신 버전
- 공식 파일 목록
- 파일 URL
- 파일 SHA-256

단, 운영 배포에서는 manifest 서명을 실제 공개키/개인키 구조로 완성해야 한다.
현재 `PUT_SIGNATURE_HERE`는 개발용 placeholder이며, 운영 보안으로 취급하면 안 된다.

## 금지 사항

런처는 다음을 해서는 안 된다.

- 사용자의 전체 PC 스캔
- 런처 인스턴스 밖의 임의 파일 검사
- 클라이언트에 private key 저장
- GitHub 토큰 저장
- Microsoft 토큰 로그 기록
- Windows Defender 또는 SmartScreen 비활성화 시도
- 보안 효과 과장

## 사용자 문구 원칙

보안 안내는 차분해야 한다.

좋은 예:

```text
공식 클라이언트 구성과 다른 파일을 정리했습니다.
```

나쁜 예:

```text
불법 파일 발견!
```

## 서버 측 강제 필요성

사용자가 일반 Minecraft 런처나 수정된 클라이언트로 직접 접속하는 상황은 클라이언트 런처만으로 막을 수 없다.

따라서 정식 버전에서는 Paper 플러그인 `ZzapchoGate`가 서버 입장 시 UUID 기반 permit을 확인해야 한다.
